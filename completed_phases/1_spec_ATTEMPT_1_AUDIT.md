File: `.github/workflows/deploy.yml` (line 8)

on:

The deploy workflow is hardcoded to trigger on the `master` branch, but GitHub has defaulted new repositories to `main` since October 2020. **ISSUE: Hardcoded default GIF tags**
File: `src/routes/slideshow/+page.svelte` (line ~202)

```js
tags: ['gif'],
```

Every GIF export is tagged with `['gif']` unconditionally. While a sensible default, tags should arguably be user-configurable at export time (or at least derived from source slides). Minor, but worth noting.

### 3. Security Basics

**ISSUE: Unescaped attribute values in SVG rendered via `{@html}`**
Files: `src/lib/annotations.js` (function `renderShapeSVG`, lines ~73–104), consumed by `src/routes/+page.svelte` (line ~136: `{@html annotationOverlaySVG(entry)}`) and `src/routes/slideshow/+page.svelte` (line ~100: `{@html annotationSVG(slides[...])}`)

The `renderShapeSVG` function interpolates shape properties directly into SVG template literals:

```js
return `<rect x="${shape.x}" y="${shape.y}" width="${shape.width}" height="${shape.height}" ${style} />`;
```

where `style` is also an interpolated string:
```js
const style = `stroke="${stroke}" stroke-width="${strokeWidth}" fill="${fill}" opacity="${opacity}"`;
```

Text content *is* properly escaped via `escapeXML`:
```js
`<text ...>${escapeXML(shape.text || '')}</text>`
```

But attribute values like `x`, `y`, `width`, `height`, `endX`, `endY`, `strokeColor`, `strokeWidth`, `fillColor`, `opacity` are not escaped. While annotations are currently self-created data stored in the user's own repo (making this a self-XSS risk), the inconsistency — escaping text but not attributes — is a defense-in-depth gap. If the annotation structure were ever populated from an external source or if an XSS in the annotation editor were discovered, the `{@html}` directive would execute arbitrary markup. Coordinate values should be coerced to numbers or escaped.

**Token storage in localStorage**: Acknowledged and documented both in the UI (lines ~55-60 of `+layout.svelte`) and in the README. The tradeoff is clearly explained. **No action needed**, this is correct for the stated single-user context.

Not a bug, but an odd inconsistency.
